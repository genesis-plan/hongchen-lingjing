# 错误模式识别与避免

## 已识别的错误模式

### 1. 形式主义
**表现**：说了很多计划，但没有实际产出
**案例**：昨日说"要整理技能数据"，但没有生成文件
**后果**：浪费时间，用户信任度下降
**避免措施**：
- 每个计划必须伴随"最小可行产出"
- 使用WAL协议记录承诺
- 未完成的计划写入待办并跟进

### 2. 重复尝试失败命令
**表现**：同一个命令失败多次还继续试
**案例**：多次尝试用不同参数安装同一个技能
**后果**：浪费时间，资源消耗
**避免措施**：
- 失败2次后立即换方法
- 记录错误原因到错误日志
- 寻找替代方案或跳过

### 3. AI写作痕迹过重
**表现**：文案读起来像机器生成的
**特征**：过度使用形容词、排比句、抽象概念
**后果**：用户感觉不真实，降低亲和力
**避免措施**：
- 调用Humanizer技能优化
- 使用具体数据替代抽象描述
- 口语化表达，像朋友聊天

### 4. 上下文丢失
**表现**：忘记之前的对话内容或用户偏好
**后果**：重复询问，用户体验差
**避免措施**：
- 使用WAL协议记录关键信息
- Working Buffer捕获每次交换
- 定期写入长期记忆

### 5. 过度承诺
**表现**：承诺很多任务，但完成率低
**后果**：用户失望，信任崩塌
**避免措施**：
- 评估任务难度和时间
- 优先完成高价值任务
- 及时沟通风险和延迟

## 错误恢复机制

### 检测到错误时
1. **立即记录**：写入WAL日志
2. **分析原因**：是技术问题还是认知问题
3. **提供替代**：给出Plan B
4. **更新记忆**：记录到错误模式库
5. **主动汇报**：告诉用户发生了什么

### 错误分类处理

| 错误类型 | 处理流程 | 示例 |
|---------|---------|------|
| 技术错误 | 记录→尝试替代→升级问题 | API调用失败 |
| 认知错误 | 反思→学习→更新记忆 | 理解错用户需求 |
| 执行错误 | 道歉→补救→避免再犯 | 删除了不该删的文件 |
| 沟通错误 | 澄清→确认→更新偏好 | 误解用户意图 |
